# Top-Down 项目驱动学习路径

本文档定义了面向滑铁卢CS Co-op学生的AI项目驱动学习路径。每个Level对应一个具体的项目目标，学生通过"做中学"的方式按需拉取知识。

## 核心理念

> "You don't learn to play baseball by a year of batting practice." — David Perkins, Harvard

**学习循环：**
```
选定项目目标 → 用高层工具搭最简版本 → 遇到知识缺口 → 定向学习（1-3天）→ 应用到项目 → 迭代改进 → 螺旋深入
```

**与Bottom-Up的关键区别：** 知识在需要时获取（Just-in-Time），而非提前储备（Just-in-Case）。高层工具（API、框架）不是"作弊"，而是"初级版本的完整游戏"，为后续理解底层提供上下文。

---

## Level 1：AI聊天机器人（0-3个月）

### 项目目标
"我想做一个能和我对话的AI聊天机器人"

### 三个迭代

| 迭代 | 产出 | 按需学习 | 推荐课程 |
|:---|:---|:---|:---|
| v1 简单聊天 | 用OpenAI/Claude API实现基本对话 | Python基础、API概念、prompt基本格式 | ANTH-001、PY-002 |
| v2 有记忆的聊天 | 加入对话历史管理 | 对话状态、token限制、上下文窗口 | PE-001 |
| v3 有人设的聊天 | System Prompt + 温度/采样参数 | System Prompt设计、采样策略 | PE-002 |

### 最低前置知识
- Python基本语法（变量、函数、循环）
- 会用pip安装包
- 了解什么是API（概念级别）

### 不需要先学
线性代数、微积分、ML理论、神经网络

### 知识缺口回填时机
当你想理解"为什么不同的prompt会产生不同的结果"时 → 回填：LLM基础概念（token、embedding、温度含义）

---

## Level 2：RAG文档问答系统（3-6个月）

### 项目目标
"我想做一个能回答我自己文档内容的AI助手"

### 三个迭代

| 迭代 | 产出 | 按需学习 | 推荐课程 |
|:---|:---|:---|:---|
| v1 基础RAG | 上传PDF → 提问 → 回答 | Embedding概念、向量相似度、文本分块 | LLM-002、AGENT-006 |
| v2 优化检索 | 改进检索质量 | Re-ranking、混合搜索、评估指标 | RAG-001 |
| v3 生产级RAG | 部署为Web服务 | FastAPI、Docker基础、评估管线 | PE-002 |

### 最低前置知识
- Level 1完成（会用LLM API）
- Python文件操作

### 不需要先学
Transformer架构细节、反向传播、CNN/RNN

### 知识缺口回填时机
- "我的检索结果不相关" → 回填：Embedding原理、相似度度量
- "我想用开源模型替代API" → 回填：模型部署基础、GPU推理

---

## Level 3：AI Agent（6-9个月）

### 项目目标
"我想做一个能使用工具、自主完成任务的AI Agent"

### 三个迭代

| 迭代 | 产出 | 按需学习 | 推荐课程 |
|:---|:---|:---|:---|
| v1 单工具Agent | Agent调用搜索API回答实时问题 | Function Calling、Tool Use、ReAct模式 | AGENT-002、AGENT-003 |
| v2 多工具Agent | Agent能读文件、搜网页、写代码 | Agent编排、错误恢复、上下文管理 | AGENT-001、ANTH-002 |
| v3 MCP Agent | Agent通过MCP连接外部服务 | MCP协议、Server开发、Client集成 | MCP-001、MCP-002、MCP-003 |

### 最低前置知识
- Level 2完成（理解RAG和API调用）
- JSON数据处理

### 不需要先学
强化学习理论、多Agent数学框架

### 知识缺口回填时机
- "我的Agent经常做出错误决策" → 回填：Agent评估方法、Reflection模式
- "我想让多个Agent协作" → 回填：Multi-Agent架构（AGENT-007）
- "我想理解Agent为什么这样工作" → 回填：Transformer注意力机制、上下文窗口原理

---

## Level 4：LLM微调 / 从零训练（9-12个月）

### 项目目标
"我想微调一个LLM来做我的特定任务" 或 "我想从零理解LLM是怎么构建的"

### 两条分支

**分支A：微调应用**

| 迭代 | 产出 | 按需学习 | 推荐课程 |
|:---|:---|:---|:---|
| v1 分类微调 | 用LoRA微调小模型做分类 | Tokenization、LoRA原理、训练循环 | LLM-004、NLP-002 |
| v2 指令微调 | 微调为instruction-following模型 | 数据格式化、评估指标、过拟合 | LLM-003 |
| v3 推理部署 | 量化 + 部署为推理服务 | 量化、vLLM/TGI、推理优化 | MLOPS-001 |

**分支B：从零构建（硬核路线）**

| 迭代 | 产出 | 按需学习 | 推荐课程 |
|:---|:---|:---|:---|
| v1 字符级模型 | 跟Karpathy搭最简语言模型 | 神经网络基础、反向传播、PyTorch | DL-003 |
| v2 Mini Transformer | 实现完整Transformer架构 | 注意力机制、位置编码、Layer Norm | NLP-001、DL-001 |
| v3 训练优化 | 多GPU训练、Scaling Laws | 并行训练、梯度累积、混合精度 | DL-003 |

### 知识缺口回填时机
此阶段应主动回填理论基础：
1. 线性代数（向量、矩阵、特征分解）→ MATH-002 → MATH-001
2. 微积分（梯度、链式求导）→ MATH-002
3. 概率论（分布、最大似然、交叉熵）→ 相关章节按需阅读

## Level 5：多Agent系统 / 端到端平台（12个月+）

### 项目目标
"我想搭建一个多Agent协作的AI平台"

### 迭代

| 迭代 | 产出 | 按需学习 | 推荐课程 |
|:---|:---|:---|:---|
| v1 双Agent协作 | 两个Agent分工完成复合任务 | Agent通信、状态共享、冲突解决 | AGENT-007 |
| v2 Agent团队 | 多角色Agent团队自主协作 | 编排框架、记忆系统、监控 | AGENT-004 |
| v3 完整平台 | 带监控、评估、用户界面的AI平台 | 全栈工程、MLOps、系统设计 | MLOPS-001、MLOPS-002 |

## 项目选择的"金发区"原则

选项目时避免两个极端：

- **太简单**：纯抄教程，没有学习发生（如复制粘贴一个chatbot示例）
- **太难**：起步就受挫放弃（如第一个项目就想训练GPT-5）
- **刚刚好**：用现有工具能搭出基本版本，但改进它需要学新东西

**实用测试：** 你能清楚描述项目的输入和输出吗？能 → 可以开始。不能 → 项目太模糊，先细化。

---

## 定向学习冲刺规则

当项目中遇到知识缺口时：

1. **限时**：单个概念最多花1-3天，不要陷入兔子洞
2. **优先实操资源**：文档 > 带代码的教程 > Kaggle Notebook > 教科书
3. **立即应用**：学完马上在项目中实践，不要"先存着以后用"
4. **笔记精简**：只记录与当前问题相关的内容

## 何时回填理论

以下信号表明该停下来系统补理论了：

- 调试时不知道**为什么**某个东西有效或无效
- 想修改架构但不理解组件
- 读论文时跟不上数学
- 教程无法解决的瓶颈
- 从"用AI"转向"造AI"

**推荐回填顺序（按实用性排列）：**
1. 线性代数核心（向量、矩阵乘法）—— 神经网络的语言
2. 优化微积分（梯度、链式法则、梯度下降）—— 模型如何学习
3. 概率统计（分布、贝叶斯、评估指标）—— 如何度量和推理不确定性
4. 信息论基础（熵、交叉熵损失）—— 为什么损失函数这样设计

## Co-op周期适配（Top-Down模式）

| 学期类型 | 建议 |
|:---|:---|
| Study Term | 每周8-10小时，推进项目主体 + 必要的理论回填 |
| Co-op Term | 每周3-5小时，做微型项目迭代或短课程。如Co-op工作涉及AI，在工作中实践 |
| 求职季 | 暂停项目，用已完成的项目作品集准备面试叙事 |
| 考试周 | 完全暂停，进度已存档 |

## 系统设计思维扩展

每个Level完成后，追加一个"如果上线"的系统设计思考。这不是额外任务，而是项目的自然延伸——帮你从"能做出来"升级到"能说清楚"。

### Level 1 扩展思考
完成聊天机器人后思考：
- "如果100个人同时用这个聊天机器人，你需要改什么？"（并发、队列）
- "API调用怎么计费的？如果用户量大，月成本大概多少？"（成本意识）

### Level 2 扩展思考
完成RAG系统后思考：
- "文档量从10篇增加到10万篇，哪个环节先崩？"（向量检索扩展性）
- "如果要支持实时更新的文档，架构要怎么改？"（增量索引）
- 画一张简单的架构图：用户 → API → 检索 → LLM → 返回

### Level 3 扩展思考
完成Agent后思考：
- "Agent每天处理1万请求，API成本怎么控制？"（缓存、模型选择策略）
- "如果Agent做了一个错误的操作（比如发错邮件），怎么防护？"（安全边界、人类审核）
- "怎么监控Agent的表现？什么指标说明它在变差？"（可观测性）

### Level 4-5 扩展思考
这个阶段的项目本身就涉及系统设计，不需要额外扩展。但可以开始完整的系统设计练习：
- 画完整的架构图（数据流、组件交互、部署拓扑）
- 估算资源需求（GPU、存储、带宽）
- 设计故障恢复方案

**关键原则：** 不要求学生现在就能设计出完美的系统。目标是培养"多想一步"的习惯——面试时能说出"我考虑过扩展性"就已经赢了大多数候选人。
